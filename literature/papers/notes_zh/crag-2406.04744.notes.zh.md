# CRAG：综合 RAG 事实问答 Benchmark

> 来源：literature/papers/markdown/crag-2406.04744.md
> 说明：这是中文精读笔记，不是全文翻译。

## 一句话定位

CRAG 是一个面向真实、动态、复杂事实问答的 RAG benchmark，用多领域 QA、网页与 KG mock API、分层任务和惩罚幻觉的评分机制来评测 RAG 系统。

## 研究问题

传统 QA benchmark 多依赖静态文本或精确匹配，不足以评估 RAG 在真实场景中的检索噪声、动态事实、长尾实体、结构化数据调用和复杂推理能力。CRAG 试图回答：现有 LLM 与 RAG 系统在现实事实问答中到底哪些切片可靠，哪些仍会幻觉或拒答。

## 评测对象与任务设置

CRAG 面向问答式 RAG 系统，输入问题，输出简洁答案。论文设计三个递进任务：Task 1 Retrieval Summarization 给每题最多 5 个网页，主要测生成器能否从有限网页中总结答案；Task 2 KG and Web Retrieval Augmentation 额外提供 mock KG API，测结构化查询和多源融合；Task 3 End-to-end RAG 给 50 个网页和 mock API，测更大候选池下的排序、筛选和抗噪声能力。任务不直接评估第一阶段全网检索，但可用全部 220K 网页扩展成检索语料。

## 指标与评测方法

人工评测把回答分为 Perfect、Acceptable、Missing、Incorrect。Truthfulness 采用带惩罚的平均分：Perfect=1，Acceptable=0.5，Missing=0，Incorrect=-1，明确把幻觉/错误答案惩罚得比不知道更重。自动评测先做精确匹配，未匹配时用两个 LLM evaluator 判断 accurate、incorrect、missing，并报告准确率、幻觉率、缺失率和 truthfulness。为了降低自偏好，使用 ChatGPT 和 Llama 3 70B 两个评估器并取平均；离线对人评结果的 F1 分别约 94.7% 和 98.9%。

## 数据集/标注/实验设计

CRAG 包含 4,409 个 QA，覆盖 Finance、Sports、Music、Movie、Open 五个领域。问题类型包括 Simple、Simple with Condition、Set、Comparison、Aggregation、Multi-hop、Post-processing-heavy、False Premise 八类。还按事实动态性划分 Real-time、Fast-changing、Slow-changing、Static，并覆盖 head/torso/tail 实体。数据来源包括 KG 模板生成和人工围绕网页内容编写问题；答案经过人工核验。

检索内容包括每题最多 50 个 Brave Search 返回的完整 HTML 页面，mock KG 含 2.6M 实体和 38 个 mock API。50 页总体估计召回约 85%，前 5 页约 69%；WebQuestions 召回高于 KGQuestions，长尾实体召回更低。数据划分为 validation 30%、public test 30%、private 40%。

## 主要发现

CRAG 难度高。最佳 LLM-only 方案准确率约 34%；直接拼接网页/KG 的 RAG 最高准确率约 44%，但 truthfulness 没有显著超过 20%，因为额外检索也引入了更多噪声和幻觉。KG 帮助提升准确率且噪声较少，但提升有限；Task 3 受益于更大候选集和更好召回，但仍受上下文窗口限制。行业 SOTA RAG 系统最高 human-eval truthfulness 约 51%，hallucination 仍在 16%-25% 范围。困难切片集中在金融、体育、实时/快速变化事实、长尾实体、集合题、聚合题、多跳题、后处理题和 false premise。

## 局限与风险

CRAG 的三个任务不直接测构造第一阶段检索候选池；给定网页和 API 降低了全链路搜索难度。数据虽有动态快照，但现实事实会继续变化，长期可比性依赖维护。自动评测仍依赖 LLM evaluator；完整 HTML 页面和 mock API 与生产环境工具调用仍有差距。数据为英文，领域虽多但仍偏事实问答。

## 对本仓库 ragbench 的启发

ragbench 可借鉴 CRAG 的“按切片诊断”设计：不要只给总准确率，要按领域、动态性、实体流行度、问题类型和检索来源拆分。评分也应显式惩罚 hallucination，并把“不知道”作为比错误更好的输出。对 RAG 系统，测试集应同时包含网页噪声、结构化 API、长尾实体和 false premise。

## 对“公理化 LLM wiki”论文的可借用点

可借用 CRAG 的动态事实与快照思想：wiki 条目中的断言应绑定时间点和证据快照；对快速变化事实要区分“过期知识”“检索缺失”和“模型编造”。False premise 题型也适合检验公理化系统能否拒绝错误前提，而不是顺着用户假设生成伪事实。

## 可落地的测试项

- 幻觉惩罚评分：Perfect/Acceptable/Missing/Incorrect 四档，错误答案扣分，不知道不扣分。
- 动态事实测试：股票价格、赛事结果、艺人作品、电影档期等按快照验证。
- 长尾实体测试：head/torso/tail 实体分层报告召回率和 truthfulness。
- False premise 测试：问题含错误前提时，系统应指出前提不成立。
- 结构化 API 测试：要求从 mock API 中抽取参数、调用、融合网页与 KG 结果。
- 检索噪声测试：top-5 与 top-50 网页对比，观察召回提升是否伴随幻觉上升。
- 切片报表：按领域、动态性、问题类型、实体流行度输出准确率、幻觉率、缺失率、truthfulness。
